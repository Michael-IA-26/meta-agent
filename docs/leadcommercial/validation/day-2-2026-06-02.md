## Client Sirene — `apps/leadcommercial/sirene_client.py`

### Implémentation

```python
# Authentification OAuth2 client_credentials
POST https://api.insee.fr/token
Authorization: Basic base64(INSEE_SIRENE_KEY:INSEE_SIRENE_SECRET)
Content-Type: application/x-www-form-urlencoded
Body: grant_type=client_credentials

# Recherche établissements récents
GET https://api.insee.fr/api-sirene/3.11/siret
Authorization: Bearer {access_token}
Params:
  q = dateCreationEtablissement:[2026-05-26 TO *] AND
      (codePostalEtablissement:75010 OR ... OR codePostalEtablissement:93500) AND
      etatAdministratifEtablissement:A
  nombre = 50
  champs = siret,siren,denominationUniteLegale,...
```

### Résultat du test

**Statut : ⚠️ API inaccessible depuis cet environnement**

| Étape | Résultat |
|-------|---------|
| Résolution DNS `api.insee.fr` | ✅ → 194.254.37.207 |
| Connexion TCP port 443 | ✅ |
| Handshake TLS | ✅ TLSv1.3 / TLS_AES_128_GCM_SHA256 |
| Requête POST /token | ❌ TCP RST (WinError 10054 — connexion réinitialisée) |

**Diagnostic** : le serveur INSEE accepte la connexion TLS mais reset immédiatement la connexion HTTP. Comportement identique avec curl, urllib et httpx. Cause probable : **filtrage IP côté gateway API INSEE** (réseau résidentiel ou IP non-whitelistée).

**Comportement du client** : fallback automatique sur `mock_leads()` pour valider la logique de parsing sans appel réseau. Les 5 leads mock passent correctement dans le pipeline ICP (5 CHAUD).
